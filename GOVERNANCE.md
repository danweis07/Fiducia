# Governance

This document describes the governance model for the Fiducia project.

## Roles

### Maintainers

Maintainers have full commit access and are responsible for releases, reviewing pull requests, and setting project direction.

### Contributors

Anyone who has had a pull request merged. Contributors are listed in the Git history and GitHub's contributor graph.

### Community Members

Anyone who participates in issues, discussions, or uses the project.

## Decision Making

- **Day-to-day decisions** (bug fixes, minor enhancements) are made by individual maintainers through the normal PR review process.
- **Significant changes** (new adapters, architecture changes, dependency additions, breaking changes) require approval from at least one maintainer and a 72-hour comment period on the PR or issue.
- **Project direction** (roadmap, governance changes, license changes) is discussed in GitHub Discussions and decided by maintainer consensus.

## Becoming a Maintainer

Contributors who demonstrate sustained, high-quality contributions and good judgment may be invited to become maintainers. The decision is made by existing maintainers through consensus.

Criteria considered:

- Consistent contributions over 3+ months
- Quality of code reviews and issue triage
- Alignment with project goals and code conventions
- Constructive community participation

## Releases

- The project follows [Semantic Versioning](https://semver.org/).
- Any maintainer can create a release.
- Breaking changes require a major version bump and a migration guide.

## Amendments

Changes to this governance document require agreement from all active maintainers and a 7-day comment period.
